SQL Analysis Queries

Production-ready queries for transaction monitoring

transactions_by_minute.sql PostgreSQL
-- Query 1: Transaction volume per minute, pivoted by status -- Used for the "Transaction Volume by Minute" chart SELECT date_trunc('minute', timestamp) AS minute, SUM(CASE WHEN status = 'approved' THEN count ELSE 0 END) AS approved, SUM(CASE WHEN status = 'denied' THEN count ELSE 0 END) AS denied, SUM(CASE WHEN status = 'failed' THEN count ELSE 0 END) AS failed, SUM(CASE WHEN status = 'reversed' THEN count ELSE 0 END) AS reversed FROM transactions WHERE timestamp >= NOW() - INTERVAL '24 hours' GROUP BY 1 ORDER BY 1;
failure_rate_monitor.sql PostgreSQL
-- Query 2: Failure rate (failed+denied+reversed) / total per minute -- Triggers alert when rate > 15% WITH minute_totals AS ( SELECT date_trunc('minute', timestamp) AS minute, SUM(count) AS total, SUM(CASE WHEN status IN ('failed','denied','reversed') THEN count ELSE 0 END) AS failures FROM transactions GROUP BY 1 ) SELECT minute, total, failures, ROUND(failures::numeric / NULLIF(total,0) * 100, 2) AS failure_pct, CASE WHEN failures::numeric / NULLIF(total,0) > 0.15 THEN 'ALERT' ELSE 'OK' END AS status_flag FROM minute_totals ORDER BY minute;
zscore_anomaly_detect.sql PostgreSQL
-- Query 3: Z-Score anomaly detection using window functions -- AVG() and STDDEV() over a 30-minute sliding window WITH rolling AS ( SELECT timestamp, status, count AS observed, AVG(count) OVER w AS rolling_mean, STDDEV(count) OVER w AS rolling_stddev FROM transactions WHERE status IN ('failed', 'denied', 'reversed') WINDOW w AS ( PARTITION BY status ORDER BY timestamp ROWS BETWEEN 30 PRECEDING AND 1 PRECEDING ) ) SELECT timestamp, status, observed, ROUND(rolling_mean, 2) AS mean, ROUND(rolling_stddev, 2) AS stddev, ROUND((observed - rolling_mean) / NULLIF(rolling_stddev, 0), 3) AS z_score, CASE WHEN ABS((observed-rolling_mean)/NULLIF(rolling_stddev,0)) > 2.5 THEN 'CRITICAL' WHEN ABS((observed-rolling_mean)/NULLIF(rolling_stddev,0)) > 1.5 THEN 'WARNING' ELSE 'OK' END AS alert_level FROM rolling WHERE rolling_stddev IS NOT NULL ORDER BY timestamp;
prometheus_alerts.promql PromQL
# Query 5: Failure rate alert — fires when rate > 15% rate(transactions_failed_total[5m]) / rate(transactions_total[5m]) > 0.15 # Query 6: Approved transaction Z-Score equivalent (Drop < -2.0) ( rate(transactions_approved_total[5m]) - avg_over_time(rate(transactions_approved_total[5m])[30m:1m]) ) / stddev_over_time(rate(transactions_approved_total[5m])[30m:1m]) < -2.0
checkout_anomaly.sql PostgreSQL
-- Query 4: POS Checkout anomaly — find hours today that deviate -- significantly from historical average (avg_last_month) SELECT time, today, avg_last_month, today - avg_last_month AS absolute_delta, ROUND((today - avg_last_month) / NULLIF(avg_last_month, 0) * 100, 1) AS pct_deviation, CASE WHEN today = 0 AND avg_last_month > 10 THEN 'CRITICAL — ZERO SALES IN BUSY HOUR' WHEN (today - avg_last_month) / NULLIF(avg_last_month,0) < -0.50 THEN 'WARNING — > 50% DROP' WHEN (today - avg_last_month) / NULLIF(avg_last_month,0) > 0.50 THEN 'WARNING — > 50% SPIKE' ELSE 'OK' END AS flag FROM checkout_data ORDER BY time;

How the Monitoring System Works

A senior engineer's explained walkthrough for new team members

📐 What is a Z-Score?

Imagine you're watching how many customers walk into a store every minute. Some minutes are busy (150 customers), some are slow (90 customers). Now, at one minute you see 250 customers. Is that unusual?

The Z-Score answers exactly that. It tells you how many standard deviations away from the average a measurement is.

z = (x - μ) / σ Where: x = current observation (e.g. 250) μ = rolling mean of last 30 mins (e.g. 120) σ = rolling std deviation (e.g. 15) z = (250 - 120) / 15 = +8.67 ← extreme SPIKE

Rule of thumb: Under a normal distribution, 95.4% of all observations fall within ±2σ. So a Z-Score beyond ±2.0 happens less than 5% of the time by chance — it's likely real.

📊 What is Standard Deviation?

Standard deviation (σ) measures spread — how much values bounce around the average. A small σ means things are steady and predictable. A large σ means things are all over the place.

σ = sqrt( Σ(xᵢ - μ)² / N ) Example values: [100, 110, 90, 105, 95] μ = (100+110+90+105+95) / 5 = 100 Σdev² = 0 + 100 + 100 + 25 + 25 = 250 σ = sqrt(250 / 5) = sqrt(50) ≈ 7.07

Why it matters: If transactions are normally 95–105/min (σ ≈ 7), then seeing 140/min is a 6σ event — almost impossible by chance. That's your fire alarm.

🪟 Rolling Window

We don't calculate σ across the whole day — we use a rolling window of 30 minutes. This means:

At any given minute, we look back at the previous 30 data points for that status, compute the mean and std dev from just those, and compare the current minute against that local baseline.

Time 10:31 → Window = 10:01–10:30 (30 points) Time 10:32 → Window = 10:02–10:31 (30 points) Time 10:33 → Window = 10:03–10:32 (30 points) ... (slides forward every minute)

This is called a sliding window and it adapts to natural time-of-day patterns (morning vs. evening traffic).

📏 Severity Tiers & Thresholds

Not every anomaly is a crisis. We tier by impact probability:

|Z| ≥ 2.5 → 🔴 CRITICAL (p < 1.2% of being noise) |Z| ≥ 1.5 → 🟡 WARNING (p < 13.4% of being noise) |Z| < 1.5 → ✅ NORMAL (within expected range) Rule-based overrides: failure_rate > 15% → WARNING failure_rate > 25% → CRITICAL approved DROP > 40% → CRITICAL (possible outage)

Hybrid approach: Statistical scoring catches gradual drift. Hard rules catch sudden cliffs (like a POS going offline). Together they minimize both false negatives (missed incidents) and false positives (alert fatigue).

🏗️ System Architecture

The system is built in C# and JavaScript:

Why C#? Strongly-typed, high-performance, identical pattern to what payment platforms like CloudWalk use for backend services. LINQ makes aggregation queries as expressive as SQL.

🧩 Algorithm Flow

Step-by-step what happens every monitoring cycle:

Step 1: Load transactions from CSV / DB Step 2: Group by status Step 3: For each status, sort by timestamp Step 4: Slide window (W=30) forward in time: a. Compute μ = mean(window) b. Compute σ = stddev(window) c. Compute z = (current - μ) / σ d. If |z| ≥ 1.5 → emit Alert Step 5: Rule check: failure_rate > 15%? Step 6: Deduplicate alerts by (timestamp, status) Step 7: Sort alerts chronologically Step 8: Serve to dashboard via JSON API

Time Complexity: O(N × W) where N = number of records and W = window size. For 26k records and W=30, this is ~800k ops — runs in milliseconds on modern hardware.
